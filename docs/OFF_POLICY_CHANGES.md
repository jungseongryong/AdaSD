# Off-Policy Self-Distillation Update

This document summarizes the code changes that add an off-policy self-distillation path to the original OPSD training code.

## High-Level Goal

The original OPSD implementation performs distribution matching on trajectories sampled from the current student policy.

In the original on-policy flow:

```text
student prompt = problem only
trajectory     = generated by the current student
teacher prompt = problem + privileged solution/context
loss           = match student and teacher distributions on the student-generated trajectory
```

The new off-policy option keeps the same student/teacher distribution matching objective, but changes where the trajectory comes from.

In the new off-policy flow:

```text
student prompt = problem only
trajectory     = loaded from the dataset
teacher prompt = problem + privileged teacher context
loss           = match student and teacher distributions on the dataset trajectory
```

The important conceptual change is that the student no longer generates the training trajectory when `--off_policy` is enabled. The trajectory is treated as fixed data.

## Why This Design

The implementation was added as an option rather than replacing the original OPSD path.

This means:

- Existing on-policy OPSD behavior is unchanged by default.
- `--off_policy` activates the dataset-trajectory path.
- The teacher context and the matched trajectory can be the same column for now.
- The teacher context and trajectory can be separated later by pointing to different dataset columns.

This is useful because the current dataset has `solution`, which can serve both as:

```text
teacher_context = solution
trajectory      = solution
```

But a future dataset can provide separate columns, for example:

```text
problem
solution
teacher_context
```

Then the trainer can use:

```bash
--teacher_context_column teacher_context
--trajectory_column solution
```

No trainer rewrite should be needed for that dataset layout.

## New CLI Arguments

The new command-line arguments are defined in `opsd_train.py`.

### `--off_policy`

Enables off-policy self-distillation.

Default:

```text
False
```

When disabled, training behaves like the original OPSD implementation:

```text
student generates trajectory -> distribution matching happens on generated tokens
```

When enabled:

```text
dataset provides trajectory -> distribution matching happens on reference trajectory tokens
```

### `--teacher_context_column`

Specifies the dataset column used as the privileged context for the teacher prompt.

Default:

```text
solution
```

This is the text inserted into the teacher prompt as privileged information.

For the current dataset, the default is enough because the available privileged context is the reference solution.

For a future dataset, this can point to a separate fixed context column:

```bash
--teacher_context_column teacher_context
```

### `--trajectory_column`

Specifies the dataset column used as the off-policy trajectory.

Default:

```text
solution
```

This is the text appended after the student prompt and teacher prompt. The loss is computed on these trajectory tokens.

For example:

```bash
--trajectory_column solution
```

means the reference solution itself becomes the trajectory for distribution matching.

## Files Changed

Three files were changed:

```text
data_collator.py
opsd_train.py
opsd_trainer.py
```

## `opsd_train.py`

`opsd_train.py` now exposes the off-policy controls through `CustomScriptArguments`.

Added arguments:

```python
off_policy: bool = field(default=False, ...)
teacher_context_column: str = field(default="solution", ...)
trajectory_column: str = field(default="solution", ...)
```

These arguments are logged to WandB:

```python
"off_policy": script_args.off_policy,
"teacher_context_column": script_args.teacher_context_column,
"trajectory_column": script_args.trajectory_column,
```

They are also passed into `OPSDTrainer`:

```python
off_policy=script_args.off_policy,
teacher_context_column=script_args.teacher_context_column,
trajectory_column=script_args.trajectory_column,
```

### Validation Added

The code blocks this combination:

```bash
--off_policy --use_tinker_loss
```

The reason is that `use_tinker_loss` uses a sampled-token policy-gradient-style objective. That objective is naturally tied to trajectories sampled from the current policy. In off-policy mode, the trajectory comes from the dataset, so the implementation currently restricts off-policy training to the full-vocabulary JSD/KL objective.

The validation is:

```python
if script_args.off_policy and script_args.use_tinker_loss:
    raise ValueError(...)
```

## `data_collator.py`

`SelfDistillationDataCollator` was extended to understand off-policy mode.

New constructor arguments:

```python
off_policy=False
teacher_context_column="solution"
trajectory_column="solution"
```

These are stored on the collator:

```python
self.off_policy = off_policy
self.teacher_context_column = teacher_context_column
self.trajectory_column = trajectory_column
```

### Teacher Context Selection

The collator now separates:

```python
teacher_context = feature.get(self.teacher_context_column, solution)
reference_trajectory = feature.get(self.trajectory_column, solution)
```

This fallback is intentional. If the requested column is not present, the code falls back to `solution`, preserving compatibility with the current dataset.

### Teacher Prompt Construction

Previously, the teacher prompt always inserted `solution`.

Now it inserts `teacher_context`.

Normal teacher prompt:

```python
f"=== Reference Solution Begin ===\n{teacher_context}\n=== Reference Solution End ===\n"
```

Reason-first teacher prompt:

```python
f"=== Reference Reasoning Start ===\n"
f"{teacher_context}\n"
f"=== Reference Reasoning End ===\n\n"
```

This is what makes future fixed-context datasets easy to plug in.

### Reference Trajectory Tokenization

When `off_policy=True`, the collator collects the dataset trajectory:

```python
reference_trajectories.append(reference_trajectory)
```

Then it tokenizes these trajectories without adding extra special tokens:

```python
trajectory_encoded = self.tokenizer(
    trajectory_texts,
    padding="max_length",
    truncation=True,
    max_length=max_trajectory_batch_len,
    return_tensors="pt",
    add_special_tokens=False,
)
```

The result is added to the batch:

```python
"reference_trajectory_ids": trajectory_encoded["input_ids"],
"reference_trajectory_attention_mask": trajectory_encoded["attention_mask"],
"reference_trajectory_lengths_per_example": torch.tensor(trajectory_lengths),
```

These tensors are consumed by `OPSDTrainer.training_step()`.

### EOS Handling

The collator appends the tokenizer EOS token to each off-policy trajectory if it does not already end with EOS:

```python
if eos and not trajectory.endswith(eos):
    trajectory = trajectory + eos
```

This makes the reference trajectory behave more like a generated completion, which normally terminates with an EOS token when generation finishes.

## `opsd_trainer.py`

`OPSDTrainer` now accepts the same three options:

```python
off_policy: bool = False
teacher_context_column: str = "solution"
trajectory_column: str = "solution"
```

It stores them:

```python
self.off_policy = off_policy
self.teacher_context_column = teacher_context_column
self.trajectory_column = trajectory_column
```

It also passes them into the default `SelfDistillationDataCollator`.

### Required Dataset Columns

`_set_signature_columns_if_needed()` now keeps the configured context and trajectory columns when the Hugging Face Trainer prunes unused columns.

Added logic:

```python
for column in [self.teacher_context_column, self.trajectory_column]:
    if column and column not in required_columns:
        required_columns.append(column)
```

This matters because without it, a future column like `teacher_context` could be removed before the collator sees it.

### Training Step Branch

The biggest behavioral change is in `training_step()`.

The trainer now sets:

```python
on_policy = not self.off_policy
```

Then the trajectory phase branches.

In on-policy mode, the original behavior remains:

```text
generate completion from student prompt
```

In off-policy mode, generation is skipped:

```python
reference_trajectory_ids = inputs["reference_trajectory_ids"]
reference_trajectory_attention_mask = inputs["reference_trajectory_attention_mask"]

generated_ids = torch.cat([inputs["student_prompts"], reference_trajectory_ids], dim=1)
generated_attention_mask = torch.cat(
    [inputs["student_prompt_attention_mask"], reference_trajectory_attention_mask],
    dim=1,
)
```

The variable name `generated_ids` is kept to minimize changes to the rest of the training path, but in off-policy mode it means:

```text
[student_prompt][dataset_trajectory]
```

not:

```text
[student_prompt][student_generated_completion]
```

### Student and Teacher Sequences

After the branch, both modes share the same sequence construction logic.

Student sequence:

```python
inputs["student_input_ids"] = generated_ids
inputs["student_attention_mask"] = generated_attention_mask
```

In off-policy mode, this means:

```text
[student_prompt][reference_trajectory]
```

Teacher sequence:

```python
teacher_full_ids = torch.cat([teacher_prompts, generation_ids], dim=1)
inputs["teacher_input_ids"] = teacher_full_ids
```

In off-policy mode, this means:

```text
[teacher_prompt_with_privileged_context][reference_trajectory]
```

The same trajectory is appended to both student and teacher inputs. This is required because the loss compares student and teacher next-token distributions over the exact same token positions.

### Labels and Masking

The label construction is mostly unchanged.

Prompt tokens are masked:

```python
labels[i, :actual_prompt_len] = -100
```

Padding tokens are masked:

```python
labels[labels == self.processing_class.pad_token_id] = -100
```

As a result, the loss is only computed on trajectory tokens.

In off-policy mode:

```text
loss tokens = dataset trajectory tokens
```

In on-policy mode:

```text
loss tokens = student-generated completion tokens
```

### Loss Function Reuse

`compute_loss()` did not need a separate off-policy implementation.

It already computes:

```python
student_logits = outputs_student.logits[:, student_prompt_len - 1 : -1, :]
teacher_logits = outputs_teacher.logits[:, teacher_prompt_len - 1 : -1, :]
```

These slices mean:

```text
take logits that predict the trajectory tokens
```

That logic works for both:

```text
student-generated trajectory
dataset-provided trajectory
```

The default loss remains:

```python
self.generalized_jsd_loss(...)
```

So off-policy self-distillation still uses token-level distribution matching between student and teacher.

## Exact Training Behavior

### On-Policy Mode

Command:

```bash
python opsd_train.py
```

or any existing OPSD script without `--off_policy`.

Behavior:

```text
1. Build student prompt from problem.
2. Build teacher prompt from problem + teacher context.
3. Student generates completion.
4. Student forward runs on [student prompt][student completion].
5. Teacher forward runs on [teacher prompt][same student completion].
6. JSD/KL loss matches distributions on student completion tokens.
```

### Off-Policy Mode With Current Dataset

Command:

```bash
python opsd_train.py \
    --off_policy \
    --teacher_context_column solution \
    --trajectory_column solution
```

Behavior:

```text
1. Build student prompt from problem.
2. Build teacher prompt from problem + solution.
3. Do not generate from student.
4. Use solution as the trajectory.
5. Student forward runs on [student prompt][solution].
6. Teacher forward runs on [teacher prompt with solution][solution].
7. JSD/KL loss matches distributions on solution tokens.
```

### Off-Policy Mode With Future Separate Context Dataset

Future dataset example:

```text
problem: "..."
solution: "full reference trajectory to match on"
teacher_context: "fixed privileged context for the teacher"
```

Command:

```bash
python opsd_train.py \
    --off_policy \
    --teacher_context_column teacher_context \
    --trajectory_column solution
```

Behavior:

```text
1. Build student prompt from problem.
2. Build teacher prompt from problem + teacher_context.
3. Do not generate from student.
4. Use solution as the trajectory.
5. Student forward runs on [student prompt][solution].
6. Teacher forward runs on [teacher prompt with teacher_context][solution].
7. JSD/KL loss matches distributions on solution tokens.
```

This is the intended path for the future fixed-context dataset.

## Relationship to SFT

Off-policy self-distillation is similar to SFT because both train on dataset reference trajectories.

However, they are not the same objective.

SFT uses hard labels:

```text
maximize log p_student(reference_token)
```

Off-policy self-distillation uses a soft teacher distribution:

```text
match p_student(. | student context)
to    p_teacher(. | privileged teacher context)
```

The target is not just the one reference token. The target is the teacher's full next-token distribution, optionally restricted by `top_k_loss` and stabilized by `jsd_token_clip`.

## Important Caveats

### Teacher Context Leakage

With the current default:

```text
teacher_context_column = solution
trajectory_column      = solution
```

the teacher sees the same solution that is later used as the matched trajectory.

This is intentional for the first implementation because it directly supports the requested behavior. But it also means the teacher has very strong privileged information.

The later fixed-context dataset should reduce or control this by separating:

```text
teacher context
matched trajectory
```

### `use_tinker_loss` Is Disabled

Off-policy mode currently supports the full-vocabulary JSD/KL objective only.

The sampled-token policy-gradient-style loss is blocked because its interpretation depends on student-sampled trajectories.

### Variable Naming

Some existing variable names such as `generated_ids` and `generation_ids` are reused in off-policy mode to keep the patch small.

In off-policy mode, these variables should be read as:

```text
selected trajectory ids
```

not literally student-generated ids.
